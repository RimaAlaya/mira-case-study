The Problem

10,000 photos, one social media agent, and not enough hours in the week

A home improvement brand already had a content automation agent in production — call her Candy. Candy writes their social media posts: catchy captions, the right hashtags, scheduled across platforms.

But Candy needed images. Polished, on-brand, publication-ready images. The brand had a Google Drive library bursting with 10,000+ raw product photos — sunrooms, pergolas, railings, installations — each one usable but not publishable. A car in the driveway here. An overcast sky there. A ceiling fan in a sunroom that needed to look airy.

The bottleneck wasn't taste. It was 10,000 photos in folders, each one needing the same five fixes, with no team big enough to do it by hand.

What was slowing them down

  • Every photo varied — sky, lighting, clutter, weather, season
  • Interior shots needed staging. Exterior shots needed environment cleanup. Different rules entirely.
  • The library kept growing — dealers and contractors uploaded new photos every week
  • Manual retouching wasn't financially possible. Outsourcing meant inconsistent style.

What they needed

  • One tool that handles the entire backlog — and every future upload
  • Scene-aware enhancement that knows interior from exterior
  • Folder structure preserved exactly — Candy expects the AI library to mirror the source
  • Resumable, crash-safe, never re-process the same image twice
  • Zero-touch operation after deployment
The Solution

Mira — the agent that makes Candy's job possible

NDA notice: client name anonymized. Architecture, code, and pipeline are original work.

Mira is a Python pipeline that walks a Google Drive source library, decides what each image needs, calls Vertex AI to actually do the work, and mirrors the result into a destination library — preserving every folder, every name, every hierarchy. Once the backlog cleared, a Cloud Run Job took over for weekly automated runs.

The team uploads new product photos. Mira polishes them by Monday morning. Candy posts them by Tuesday afternoon. Nobody touches a thing in between.

  1. Walk the source library

    Recursive traversal of the Drive tree. Build the work queue. Skip excluded folders (drop zones, drafts). Only process real image formats.

  2. Classify the scene

    Gemini 2.5 Flash decides interior or exterior in ~3 seconds per image. The cheap model is enough — composition is all we need at this stage.

  3. Assemble the right prompt

    Interior prompts stage furniture and clear ceilings. Exterior prompts preserve composition and clean the environment. Both refuse to block doors or add ceiling fans. Non-negotiable brand rules.

  4. Queue for batch processing

    Each image + prompt is packaged into a Vertex AI batch job — exactly half the cost of real-time for the same model. The image is recorded in the manifest as "queued" the moment it enters the queue.

  5. Submit & poll

    Vertex handles the heavy lifting server-side. The script polls every 60 seconds. Large libraries get chunked into ~1,000-image batches to stay under Vertex's 24-hour-per-batch hard ceiling.

  6. Mirror to destination, update manifest

    Each enhanced image lands in the destination subfolder that mirrors its source path. The manifest entry flips from "queued" to a real output file ID. Done. Future runs skip it automatically.

Engineering

What I actually built

The system is a modular Python pipeline — nine source files, each with a single, boring responsibility. The orchestrator doesn't know what Drive's API looks like. The classifier doesn't know what a batch is. Each layer is independently testable, swappable, and (most importantly) debuggable at 3am when something breaks.

The classifier — two stages, one cheap one optional

Stage one: scene classification via Flash. Cheap, fast, accurate enough. Stage two (optional): a quality gate via Pro asking "does this image actually need enhancement?" Active during testing, disabled for the bulk run after the math showed the gate cost more than the savings.

The state tracker — manifest in Drive, not in a database

Every destination subfolder carries a _candy_manifest.json with every image processed: source ID, MD5, timestamp, model used, output ID. State lives next to the data. No external database to maintain, no sync issues, no infra to provision. A crash mid-run is fully recoverable — the next invocation reads the manifest and skips what's done.

The batch editor — abstracting Vertex's rough edges

Vertex batch prediction has quirks. A hard 24-hour deadline per batch. A strict JSONL schema where the labels field must be a flat string, not a dict (don't ask me how I learned that). All-or-nothing result release. The batch editor handles all of it — the orchestrator just queues items, waits, gets results back.

War Story

When 10,422 images timed out at hour 23:59

The first batch ran the full 24 hours and completed only 1,345 of 10,422 images before Vertex's deadline killed the job. 9,077 incomplete. No money lost (Vertex only charges for successful predictions), but a lot of preprocessing work hung in the balance.

Recovery: a harvest script pulled the 1,345 successes from GCS into Drive. A chunked recovery script picked up the remaining 9,077 from the manifest, split them into 10 batches of ~1,000 each, and ran them sequentially. Each chunk finished comfortably under 24 hours. Total recovery: ~7 days of unattended runs. Zero re-preprocessing. Zero re-billing.

Reflection

Four things this didn't teach me, it slapped me with

State belongs next to the data

The single best decision was putting the manifest inside each destination subfolder, not in a database. Drive was already the source of truth for files. Adding a separate database for "which files are done" would have been one more thing to keep in sync. With the manifest in Drive, state is trivially backed up, trivially inspectable, and trivially survives any infra change.

Record state on intent, not on completion

An early version recorded an image as "done" only after the batch finished. When a batch timed out mid-run, thousands of preprocessing calls were lost — the next run would redo them. The fix was tiny: record images as "queued" the moment they entered the batch, with a status flag. A crash mid-batch now loses zero preprocessing work.

Cloud quotas are the real design constraints, not the model

The model can do almost anything. The platform can't. Vertex's 24-hour batch deadline, Drive's API rate limits, Gemini Pro's per-minute quotas — these were the actual design constraints. Building around them mattered more than tuning the prompt.

Cheap pre-filters can cost more than they save

The quality gate — a Pro model call asking "does this need enhancement?" — looked smart on paper. In testing, only 5% of images were marked publication-ready. The math: the gate cost more per image than the savings from skipping 5%. Removed for the bulk run. Re-enabled for the weekly trickle where the dynamics shift.
